## 一、项目起源：从骨架到世界观

### 对话实录

用户最初提供了一份骨架大纲 `skeleton.txt`，描述了一个被困在时间裂隙中的意识体——叙述者，他以第三人称视角跟随勇者米迦经历五次轮回。每次轮回以黎明苏醒开始，黄昏刺杀事件结束。叙述者的真实身份是魔王，但他自己并不知道。

### 我的思考过程

收到骨架后，我做的第一件事是确定技术栈。视觉小说有几种主流选择：

- **Ren'Py**：Python 语法，社区成熟，内置存读档/CG回顾/选项分支，跨平台发布
- **TyranoBuilder**：可视化编辑器，但脚本语言封闭
- **自研引擎**：灵活但开发周期长

选择了 Ren'Py，因为它最适合"先写故事，后加美术"的开发流程。项目不需要精美立绘就能跑起来，这对原型阶段至关重要。

### 核心设定确立

在骨架基础上，我完善了以下设定：

**属性系统**设计了五个维度——警觉、执念、同步、权能、meta_awareness。前四个是传统TRPG属性，第五个是这个项目独有的：叙述者对自身"被书写"状态的认知程度。这个属性是通往隐藏结局（飞升结局）的钥匙。

**暗骰机制**的灵感来自桌游中的DM暗骰：当玩家的 meta_awareness 累积到 2 以上时，每次判定会额外暗投一枚 d20。若暗骰命中 20，触发叙述者突破第四面墙的隐藏结局。玩家看不到暗骰的数值，只能感受到"有什么东西变了"。

**两个关键NPC**：
- **米迦**：紫发，嬉皮笑脸，武器是一把缺口严重的锈剑。每次轮回都死于黄昏刺杀。他的笑容下面藏着什么，叙述者五次轮回都没能看清。
- **塞西尔**：刺客之一，金发，冷峻疲惫。他的家族曾看守旧神殿。在第四次轮回被招募进队伍，成为连接"过去"与"现在"的桥梁。

### 项目文件结构

```
my-vn/
├── game/
│   ├── script.rpy          # 主游戏脚本（Ren'Py格式）
│   ├── screens.rpy          # UI界面定义
│   ├── options.rpy          # 项目配置
│   └── gui.rpy              # GUI样式
├── text_runner.py           # 终端文字版（用于快速测试叙事）
├── story_text.txt           # 纯文本版故事内容
├── branch_tree.html         # 可视化分支树（浏览器打开）
└── skeleton.txt             # 原始骨架大纲
```

四文件并行维护的架构从一开始就确立了——script.rpy 是正式引擎版，text_runner.py 用于终端快速测试叙事节奏，story_text.txt 是纯文本参考，branch_tree.html 是浏览器可视化分支图。任何叙事修改都必须同步到四个文件。这个决策后来证明是正确的，但也大大增加了每次修改的工作量。

## 二、分支设计：四选一到六选一的演进

### 初始设计：每轮回四选一

最初设计了 5 个轮回，每轮回 4 个选项，共 20 条路径，导向 6 个结局。

### 用户的扩展要求

用户看了初始方案后，要求将每个轮回的选项扩展到 5-6 个。理由是：4 个选项太"整齐"，缺少那种"我不知道该选什么"的焦虑感。

### 我的思考过程

扩展选项不仅仅是加一行文字。每个新选项需要：
1. **叙事合理性**：它必须在当前轮回的情境下说得通
2. **属性差异化**：它赋予的属性组合必须和已有选项不同
3. **叙事独创性**：它导向的文本不能和已有路径重复

我为每个轮回设计了新选项的核心理念：

- **Loop 1 新增"捡起水壶递过去"**：这个选项的灵感来自"最小的善意"。在哥特暗黑的氛围中，一个递水壶的动作比任何豪言壮语都更有力量。它给予同步属性和米迦的信任。
- **Loop 2 新增"你剑上的锈——不是普通的锈"**：引入铁匠铺/旧神殿的伏线。叙述者通过观察锈迹，暗示他拥有超越轮回的记忆碎片。
- **Loop 3 新增"你做过梦吗？"和"翻进铁匠铺搜查"**：前者是情感线（通过梦境暗示轮回），后者是探索线（直接搜查获得权能）。
- **Loop 4 新增"你还记得那条黑水河吗？"和"检查铁匠铺地窖"**：黑水河是叙述者与米迦前世的共同记忆，地窖藏着旧神殿的秘密。
- **Loop 5 新增"你还记得那个河对面的人吗？"和"带塞西尔一起去旧神殿"**：前者是身份揭示的终极选项，后者引入塞西尔的家族记忆线。

### 最终数据

| 轮回 | 选项数 | 主题 | DC | 判定属性 |
|------|--------|------|-----|---------|
| Loop 1 | 5 | 初遇 | DC12 | 警觉 |
| Loop 2 | 5 | 重逢 | DC12 | 执念 |
| Loop 3 | 6 | 觉醒 | DC12 | 同步 |
| Loop 4 | 6 | 沉沦 | DC12 | 执念+同步 |
| Loop 5 | 6 | 终局 | DC12 | 执念+同步 |

总计 **28 条路径**，导向 **6 个结局**。

### 六结局设计

| 结局 | 触发条件 | 核心意象 |
|------|---------|----------|
| 黎明 (DAWN) | 最终判定成功，mika_knows_me = False | 两个陌生人在晨光中并肩 |
| 相知 (KNOWING) | 最终判定成功，mika_knows_me = True | 身份揭示，真相被接受 |
| 飞升 (ASCENDING) | 暗骰命中20 | 叙述者突破第四面墙 |
| 铁锈 (RUST) | 最终判定失败，mika_trust < 2 | 米迦独自面对命运 |
| 裂痕 (FRACTURE) | 最终判定失败，loop_count ≥ 3 | 意识碎裂，轮回永续 |
| 余烬 (EMBER) | 最终判定大失败(d20=1) | 废墟中等待第六次轮回 |

## 三、试玩方案：终端版与引擎版并行

### 用户的请求

用户说："现在我要试玩一下，给我试玩方案。"

### 我的思考过程

Ren'Py 引擎版需要完整的引擎环境（用户需要下载 Ren'Py SDK），而且每次修改脚本都需要重启引擎。对于叙事密集型的原型阶段，这太慢了。

我提出了两个方案：

**方案一：终端文字版 (text_runner.py)**
- 用纯 Python 实现，不需要任何额外依赖
- 在终端/PowerShell 中直接运行 `python text_runner.py`
- 通过 `input()` 实现选项选择，`time.sleep()` 控制叙事节奏
- 适合快速迭代叙事内容

**方案二：Ren'Py 引擎版**
- 需要下载 Ren'Py SDK（用户后来放在了 `D:\renpy\renpy-8.5.2-sdk\`）
- 通过 Ren'Py 启动器运行
- 有完整的 GUI、存读档、CG回顾
- 适合最终发布

用户两个都要。于是我先完成了 text_runner.py 的全部叙事内容，再同步到 script.rpy。

### text_runner.py 的设计

封装了三个核心输出函数：

```python
def narrator(text):
    """叙事者文本——无名字"""
    print(f"\n  {text}")
    time.sleep(0.3)

def say(speaker, text):
    """角色对话——带名字"""
    print(f"\n  [{speaker}] {text}")
    time.sleep(0.3)

def press_enter():
    """暂停等待玩家输入"""
    input("\n  按回车继续...")
```

骰子系统用 Python 的 `random.randint(1, 20)` 实现，和 Ren'Py 版的 `renpy.random.randint(1, 20)` 逻辑完全一致。

## 四、文风定调：坡与勃朗特的融合

### 用户的核心要求

用户对文风有极其明确的定位：**爱伦·坡的心理恐怖** 加上 **艾米莉·勃朗特的野性浪漫**。

这不是简单的"黑暗风格"。用户要求的是一种特定的文学质感：

**坡的贡献**：
- 紧张/惊骇时刻切换为断奏节奏（短句、碎句、感叹号）
- 通过重复模拟执着与痴迷
- 不可靠叙述者的偏执视角

**勃朗特的贡献**：
- 场景铺垫和内心独白使用绵长盘旋的句子
- 模仿荒原上呼啸的风声
- 氛围浓烈，情感暴烈到近乎暴力

### 用户原话（大意）

> "叙述者对米迦的感情不是温柔的守护，而是一种暴力的、寄生的、无法拔除的执念。每次描写米迦死亡的场景，要让读者感受到叙述者的皮肤在灼烧、骨头在断裂。"

### 我的思考过程

这个文风要求意味着我不能用普通的"网文腔"或"轻小说腔"来写。我需要：

1. **句式错落**：长短交替，不能每句都是"主谓宾"的平铺
2. **感官过载**：每个抽象概念都必须附着在一个具体的感官细节上——不是"他很悲伤"，而是"他的指甲嵌进了掌心，血珠沿着掌纹渗出来，像一条蜿蜒的红色河流"
3. **欧化中文**：模仿19世纪翻译文学的语法结构，从句套从句，修饰语层层叠叠
4. **不可靠叙述**：叙述者看到的不一定是真实的，他的感知被轮回和执念扭曲

### 选定的风格参考

用户明确提到了一个风格锚点："模仿 DeepSeek 2025 年初的写作风格"——即那种冷峻、克制、偶尔爆发的中文文学质感。

## 五、第一轮文风清洗：57处违规

### 用户的禁令清单

用户给出了一份极其详细的禁令清单，分为**禁用句式**和**禁用词汇**两类：

**禁用句式**（完整列表）：
- "不是……而是……"
- "在……的过程中"
- "总而言之""综上所述"
- "不仅……更……"
- "不是……是……是……"
- "不是……不是……是……"
- "是……的基石/关键/必修课"
- 排比句、对偶句、反复等修辞性排叠结构
- "不x，不y，不z，就"格式
- "这就够了"
- "很…，但很…"
- 总-分-总结构

**禁用词汇**：兜住、接住、稳、守、极其

**替换原则**：每个违规句必须替换为一个具体的感官画面，而不是换一个同义词。

### 我的思考过程

我逐文件扫描了所有文本，用正则表达式匹配禁用句式，然后手动审查每一个匹配项。关键的挑战在于：替换不能只是"换个说法"，而必须是一个全新的感官画面。

**举例**：

原文：
> "他不是害怕，而是愤怒。"

替换为：
> "他的手指在颤抖。不是恐惧的那种颤抖——是琴弦绷到极限时发出的那种嗡鸣，从指尖传到手腕，传到肘关节，传到肩胛骨。"

原文：
> "在战斗的过程中，他逐渐明白了真相。"

替换为：
> "剑刃劈开空气的声音在他耳边炸开。第一下。第二下。第三下。每一次金属碰撞都在他的颅骨内侧留下一道刻痕，直到那些刻痕连成了一句话。"

### 修复统计

| 文件 | 违规处数 | 修复方式 |
|------|---------|---------|
| script.rpy | 20处 | 全部替换为感官画面 |
| text_runner.py | 17处 | 同步替换 |
| story_text.txt | 20处 | 同步替换 |
| **合计** | **57处** | |

## 六、第二轮文风清洗：破折号与AI腔

### 用户的第二轮要求

在第一轮修复完成后，用户提出了更严格的要求：

> "——这种AI文风明显的符号在所有的文本里3000字内最多出现1次。同时检查禁用词汇频率。"

### 我的思考过程

这个要求让我意识到一个问题：破折号（——）是AI生成中文文本时最明显的"指纹"之一。人类写作者很少在一段文字中密集使用破折号，但AI模型（包括我自己）倾向于用破折号来插入补充说明、制造戏剧性停顿。

我统计了各文件的破折号数量：
- script.rpy: **255个**
- text_runner.py: **156个**
- story_text.txt: **249个**
- branch_tree.html: **37个**
- **总计: 697个**

这个数字触目惊心。按照3000字1个的标准，22000字的script.rpy最多只能有7个破折号。

### 第二轮新增禁令

- 破折号（——）3000字内最多出现1次
- 禁止无意义的精确数字（"零点二秒"、"两毫米"——这种数字暗示了一种虚假的精确感）
- 每500字内"不是"不超过3次
- 避免高频使用"首先、其次、然后、最后"等程式化连接词
- 减少重复说法和同义句堆叠

### 修复策略

对于破折号的处理，不能简单地删除。每个破折号都有其叙事功能——停顿、转折、补充说明。我需要为每个破折号找到替代方案：

- **补充说明型破折号** → 改用从句或括号性插入语
- **戏剧停顿型破折号** → 改用句号断句或换行
- **转折型破折号** → 改用"但""然而"或直接用新段落

### 第一轮文风修复后的统计

修复了约57处禁用句式违规后，进入了破折号专项清洗阶段（详见第九节）。

## 七、命运丝线：从机械骰子到氛围叙述

### 用户的请求

用户看了骰子判定的显示效果后说：

> "如果有暗骰，显示为命运的四线在此刻波动之类的句子，要有逼格。"

### 我的思考过程

原来的骰子显示是这样的：
```
【d20=14 + 警觉(3) = 17 ≥ DC12】
```

这是标准的TRPG格式，准确、清晰，但完全破坏了哥特暗黑的氛围。在一轮沉浸式的心理恐怖叙事中，突然跳出一行机械数据——就像在看恐怖电影时屏幕下方弹出一行Excel表格。

我设计了**命运丝线**隐喻：骰子不是冰冷的随机数，而是命运织机上颤动的丝线。每次判定都是丝线的一次颤动——收紧、松弛、绷紧、断裂。

### 命运丝线的五种状态

**成功**：
> 「命运的丝线在此刻微微收紧，像琴弦被拨动了一下。」

**失败**：
> 「命运的丝线微微松弛。方向偏了。」

**暴击（d20=20）**：
> 「命运的丝线在此刻骤然绷紧，发出一声无人听见的嗡鸣。」

**大失败（d20=1）**：
> 「命运的丝线骤然断裂。碎裂的声音从很远的地方传来，像冰面下的河水突然涌出。」

**暗骰触发**：
> 「……有什么东西变了。一道裂缝。在那片虚无的最深处，出现了一道不该存在的裂缝。」

### 技术实现

**Ren'Py 版**：创建了 `fate_notify` screen，显示在屏幕顶部（yalign 0.08），3秒后自动消失。使用斜体、低饱和度颜色（#8a7a6a），确保它像一个幽灵般的提示而非弹窗。

```renpy
screen fate_notify(message, roll_text=""):
    zorder 200
    timer 3.0 action Hide("fate_notify")
    frame:
        xalign 0.5
        yalign 0.08
        xpadding 24
        ypadding 12
        background None
        vbox:
            spacing 4
            xalign 0.5
            text message size 16 color "#8a7a6a" xalign 0.5 italic True
            if roll_text:
                text roll_text size 12 color "#5a4a3a" xalign 0.5
```

**text_runner.py 版**：直接用 `narrator()` 输出氛围文字，在输出前加 0.8 秒延迟模拟"命运的凝视"。

**story_text.txt 版**：用 `[命运的丝线...]` 标记替代原有的 `[DICE]` 标记。

### 原始骰子代码

```python
init python:
    def roll_check(stat_name, stat_value, difficulty):
        dice = renpy.random.randint(1, 20)
        total = dice + stat_value
        if dice == 20:
            return (dice, stat_value, total, True, "critical_success")
        elif dice == 1:
            return (dice, stat_value, total, False, "critical_failure")
        success = total >= difficulty
        result_type = "success" if success else "failure"
        return (dice, stat_value, total, success, result_type)

    def meta_roll():
        return renpy.random.randint(1, 20)
```

骰子判定的结果类型（critical_success / critical_failure / success / failure）决定显示哪条命运丝线描述，同时决定走向哪个叙事分支。

## 八、四线变丝线：用户的一声怒吼

### 事件经过

在替换骰子标记时，我犯了一个低级错误：把"命运的丝线"写成了"命运的四线"。

用户原话：

## 十、PowerShell编码灾难：一个惨痛的教训

### 事故经过

在修复"四线"→"丝线"的过程中，我使用了 PowerShell 的 `-replace` 操作符配合 `-NoNewline` 参数来批量替换文件内容。这是一个致命的错误。

### 根因分析

PowerShell 的 `Set-Content` 和 `-replace` 操作符**默认使用系统编码**（在中文 Windows 上是 GBK/CP936），而不是 UTF-8。当它写入一个 UTF-8 编码的文件时：

1. 读取阶段：PowerShell 以 GBK 编码读取 UTF-8 文件，部分多字节 UTF-8 序列被错误解析
2. 写入阶段：PowerShell 以 GBK 编码写回，进一步损坏已错误解析的字符
3. 具体损坏：
   - 所有 `"` (U+201D) 右弯引号被剥离（GBK 编码中不存在此字符）
   - `"他" "` 对话格式损坏为 `"。 "`（引号丢失后，残余字符重新组合）
   - 大量中文字符被替换为 `�` 乱码
   - 文件从约 975 行膨胀到 1985 行（每行后多出空行）

### 损坏范围

- **script.rpy**：100+ 处对话格式损坏，文件末尾（第1105-1112行）出现严重乱码
- **text_runner.py**：文件末尾（第585行之后）出现乱码
- **story_text.txt**：文件末尾（第660行之后）出现乱码
- **branch_tree.html**：损坏最轻，大部分内容完好

### 修复过程

编写了多个 Python 修复脚本，按顺序执行：

1. **fix_quotes.py** — 恢复缺失的闭合引号
2. **fix_speaker.py** — 修复损坏的说话人标签（`"。 "` → `"他" "`）
3. **fix_all.py** — 综合修复脚本
4. **repair.py** — 引号匹配修复
5. **fix_dialogue.py** — 批量修复100处损坏的对话格式
6. **fix_blanks.py** — 清理多余空行

### 修复的思考过程

最棘手的问题是：损坏的文本中，哪些字符是"原文"，哪些是"乱码"？没有版本控制的历史记录，我只能：

1. 对比四个文件中同一段落的上下文
2. 根据 Ren'Py 语法结构推断正确的标签格式
3. 根据叙事逻辑重建被损坏的段落

例如，`"。 "text"` 这种格式显然是 `"他" "text"` 的损坏——因为 `"。 "` 不是任何合法的 Ren'Py 说话人标签。通过正则匹配 `"。 "` 并结合上下文判断说话人，批量修复了 100 处。

### 教训

**永远不要用 PowerShell 的原生 cmdlet 操作 UTF-8 中文文件。** 所有后续的文件操作都改用 Python 脚本完成。

用户保留了所有修复脚本，放在项目根目录下，作为"事故现场"的证据。

## 十三、核心设计理念

### 1. 不可靠叙述者

玩家扮演的"我"是魔王，但他的记忆被五次轮回碾碎。他不确定自己的感知是否真实——他看到的米迦是真的米迦，还是他执念的投影？他感受到的温暖是真的温暖，还是深渊在伪装？

这种不可靠性不仅体现在叙事文本中，也体现在骰子系统中：同一个选项，不同掷骰结果会导向截然不同的叙事，而叙述者会为每一种结果找到"合理"的解释。

### 2. 寄生式情感

叙述者对米迦的感情不是温柔的守护，而是一种暴力的、寄生的、无法拔除的执念。每一次描写米迦死亡的场景，叙述者的身体都会有物理反应——皮肤灼烧、骨头断裂、指甲嵌进掌心。这不是"悲伤"，这是一种寄生虫在宿主体内翻滚的痛感。

### 3. 感官优先

每个抽象概念都必须通过具体的感官细节传达。禁用"悲伤""恐惧""愤怒"等抽象情感词，取而代之的是：

- 悲伤 → "他的指甲嵌进了掌心，血珠沿着掌纹渗出来"
- 恐惧 → "他的脊椎像一根被冻住的琴弦，从尾椎到颈椎都在发出细微的嗡鸣"
- 愤怒 → "他的手指在颤抖。不是恐惧的那种颤抖——是琴弦绷到极限时发出的那种嗡鸣"

### 4. 命运的丝线

骰子不是冰冷的数字，而是命运织机上颤动的丝线。这个隐喻贯穿整个游戏：每次判定都是丝线的一次颤动，暴击是丝线骤然绷紧，大失败是丝线断裂。暗骰触发时，玩家看不到数值，只能感受到"有什么东西变了"。

### 5. 渐进揭示

玩家（叙述者）对自身身份的认知随轮回加深：

- **Loop 1-2**：旁观者——"我在看一个人的命运"
- **Loop 3-4**：参与者——"我在改变一个人的命运"
- **Loop 5**：被书写者——"我发现自己的命运也被书写着"
- **ASCENDING结局**：书写者——"我突破了纸页，看到了正在写这个故事的你"
