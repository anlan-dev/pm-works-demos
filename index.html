🐾 宠伴·健康台账

通过用户研究、假设验证与持续迭代，解决多宠家庭「疫苗遗忘」与记录摩擦，推动核心行为指标提升。

1｜问题定义：谁在痛、痛在哪

通过对 20 位宠物主的深度访谈与竞品评论阅读：多宠家庭中约七成在月度尺度上至少发生过一次「接种相关遗忘」；共性痛点不是「晒宠社交」，而是简单、免费、无需学习成本的提醒与记录。据此将产品定位为轻量健康记录册，并砍掉社区、商城等非核心模块，聚焦多宠档案、疫苗日历、花费看板与智能提醒四条主线。

2｜关键决策与验证

OCR 做不做、怎么做：原型测试中「手动录入疫苗信息」劝退率最高。我横向对比三种方案，从隐私、成本、准确率评估后选择本地化 OCR——访谈里约 83%的用户明确表示不接受把接种凭证上传公有云，决策首要因素是隐私信任，而非单纯成本。

提醒机制是否重构：假设「按宠定制 + 灵活忽略」比单一开关更降低打扰感、提升配置意愿。分组对比后，实验数据表明「开启提醒设置」相关意愿由约 26% 提升至约 42%，7 日回访意愿由约 31% 提升至约 44%（与下图一致）。决策：全量引入灵活提醒，并用「稍后 / 今日忽略」平衡打扰与留存。

SQL 口径自检（点击展开）

我会用与图表叙事一致的事件表反推核心指标，验证指标定义 / JOIN 关系 / GROUP BY 粒度可复算，确保「转化率、7 日留存」口径一致可解释。

转化率 SQL 7日留存 SQL 事件表口径校验 

SELECT group_type, COUNT(DISTINCT user_id) AS total_users,
       COUNT(DISTINCT CASE WHEN event_name='set_reminder' THEN user_id END) AS converted_users
FROM experiment_events
WHERE date BETWEEN '2026-02-01' AND '2026-02-14'
GROUP BY group_type;
-- D7 留存：D0 首开用户 LEFT JOIN 第7天活跃用户
查看完整 SQL 
-- 转化率：按实验组统计「设置提醒」行为
SELECT
    group_type,
    COUNT(DISTINCT user_id) AS total_users,
    COUNT(DISTINCT CASE WHEN event_name = 'set_reminder' THEN user_id END) AS converted_users,
    ROUND(COUNT(DISTINCT CASE WHEN event_name = 'set_reminder' THEN user_id END) * 1.0 / COUNT(DISTINCT user_id), 2) AS conversion_rate
FROM experiment_events
WHERE date BETWEEN '2026-02-01' AND '2026-02-14'
GROUP BY group_type;

-- 7 日留存匹配
SELECT
    a.group_type,
    COUNT(DISTINCT a.user_id) AS d0_users,
    COUNT(DISTINCT b.user_id) AS d7_users,
    ROUND(COUNT(DISTINCT b.user_id) * 1.0 / COUNT(DISTINCT a.user_id), 2) AS d7_retention
FROM
    (SELECT DISTINCT user_id, group_type FROM experiment_events WHERE date = first_open_date) a
LEFT JOIN
    (SELECT DISTINCT user_id FROM experiment_events WHERE date = DATE_ADD(first_open_date, INTERVAL 7 DAY)) b
ON a.user_id = b.user_id
GROUP BY a.group_type;

4｜反馈驱动的迭代与结果

上线后跟踪 5 位种子用户的行为与主观反馈，识别「算下次日期」「单月花费看不清」等高频摩擦。迭代下拉选苗自动算有效期与按月聚合花费视图后，单次记录路径的用户主观反馈为操作耗时约缩短 60%。在零服务器成本前提下完成「想法 → 验证 → 迭代 → 可安装 PWA」，并沉淀完整复盘文档。

若再做一次，我会更早用纸质流程测多宠提醒冲突；下一轮优先验证「复诊时一页摘要给医生」是否比完整表格更能驱动留存。

🎙️ 会议·反刍舱

把「会后失忆」变成可追踪的待办闭环——用时间节省与漏项下降定义成功，而不是罗列技术栈。

1｜问题定义

访谈在职 PM / 项目协作伙伴：会后整理纪要、从群聊里打捞 action动辄半小时量级，且「会上说过、会后没落地」反复发生。核心矛盾不是「缺录音」，而是纪要复述成本高 + 待办不可追踪。

2｜关键决策

不做大而全协作套件；先做「单场会议的产出可被结构化」——录音 / 上传 → 转写 → 待办字段 → 状态流转 → 周报复盘。对隐私敏感团队保留本地 Whisper路径；结构化环节用多轮 Prompt 与漏项抽检迭代——评测标准是「待办是否覆盖真实协作语境」，而非模型名称。

3｜验证与数据

定性：个人基线下会后整理从约 30 分钟收敛到约 5 分钟可复盘。定量叙事：关注7 日待办完成率在「反刍提醒 / 状态流」等迭代节点前后是否抬升（下图与陈述同一口径）。

✅ 待办完成率趋势 · 会议反刍舱
上线前后对比｜与产品迭代节点对齐
读图要点：趋势是否与我推出的「待办状态流 / 反刍提醒」等动作同期改善，再决定下一轮优化聚焦漏项类型还是协作视图。

4｜反思与下一步

若再做一次，会更早为跨会议阻塞项单独做视图，而不是先把导出样式做满；可选 server/ 仅作为团队私有化时的增强，避免绑架默认体验。

💡 从想法到可用：我的通用方法

承接宠伴、会议等案例——这里只补一层「我通常怎么挖场景、敢砍什么、怎么收反馈」

①

🔍 先找到真正烦人的小场景

四个作品对应四类高频抱怨：养宠漏针、创作资料散、剧本改戏断层、会后待办沉底。做法仍是：聊 + 看差评，收敛到「简单、最好免费、帮我记牢」。

②

🎯 砍到最小闭环

不做社交/商城。宠伴只做记录—提醒—回看；创作舱绑定大纲与执行；剧本舱绑定修改原因；会议舱绑定转写—待办—周报。

📄 PRD 与交互说明样例

宠伴「提醒设置」模块 PRD 摘录（v1.6 灵活提醒）。结构：目标与范围 → 名词定义 → 页面状态 → 规则与边界 → 主路径流程，便于评审一次性对齐字段与状态机。

🐾 模块：提醒设置 · 疫苗 / 驱虫 · 按宠独立

文档用途：评审钉版 · 读者：产品 / 设计 / 研发

1. 背景与目标

  • 问题：多宠家庭易混淆「谁该打哪一针」；通用日历难以表达疫苗 / 驱虫各自周期。
  • 目标：在宠物维度维护待提醒事项，支持打扰可控（稍后 / 忽略）与按类关闭；状态可映射到可实现的数据结构与推送策略。

2. 范围与非目标

  • 本期包含：疫苗、驱虫两类提醒的配置、列表、推送与应用内提示；与档案计划日联动。
  • 本期不做：社交分享、医院挂号对接、跨账号家庭协作（可记入 backlog）。

3. 名词定义

术语 定义
提醒周期 某一宠物、某一类型事项从当前计划日起至下一次同类计划生成之间的窗口；驱虫可按药品周期，疫苗可按剂次。
稍后 自点击时刻起延迟打扰：2 小时后再次触发同一条目；同一条目连续「稍后」≤ 3 次（与下文规则一致）。
忽略 在当前提醒周期内不再弹出该条目；进入下一周期后重新按规则计算（不因本次忽略而永久关闭）。
关闭该类提醒 关闭该宠物下一整类通道（如「驱虫」）：不再推送，历史与手动入口保留。

4. 页面状态表

状态 进入条件 界面要点 可用操作
正常 · 有待处理 ≥1 只宠物，且存在未完成且未在本周期内被忽略至静默的提醒 按宠 Tab / 卡片；字段含类型、计划日、来源（手动 / OCR） 稍后 / 忽略 / 进详情 / 关闭该类提醒
无宠物 档案宠物数 = 0 不展示提醒配置；空状态 +「添加宠物」主按钮 添加宠物
本期无可打扰项 当前筛选维度下，事项均已完成、已改期，或本周期内已全部「忽略」 说明性空状态；提供「历史提醒」「重置本期忽略」（若产品开启） 查看历史 / 重置忽略 / 添加新计划
超期未处理 计划日 < 今日且未完成 超期标签 + 列表置顶；首页轻量提示「有超期项」 补录完成 / 改期 / 忽略本期
离线 / 同步异常 无网或备份接口失败 顶部弱提示「更改已保存在本机」 本地可继续编辑稍后 / 忽略；恢复后同步（见边界）

5. 核心交互规则

  • 稍后：点击起算 +2h 再次提醒同一条目；连续「稍后」第 4 次起不再延时，改为引导文案：请在详情中改期或标记完成。
  • 忽略：仅静默当前周期内该条推送；下一周期（如新计划日生效）重新参与排序与触发。
  • 关闭该类提醒：粒度到「某宠 + 某类」，不影响其他类型提醒；关闭后可通过设置页重新打开。

6. 异常与边界

  • 删除宠物：该宠下待弹出 / 未读提醒迁入「历史提醒」归档，主列表不再展示，避免误删纠纷。
  • 多宠同时触发：排序键 = 距离计划日的剩余天数升序；< 7 天条目优先；单次推送最多 3 条，其余进入队列顺延。
  • 离线写入冲突：稍后 / 忽略 / 开关状态本地持久化；与服务端对齐时以最近客户端操作时间为准（若启用云备份）。

7. 主路径流程

入口：系统推送 或 应用内 Banner / 列表红点 → 用户打开提醒卡片 ├─ 「稍后」→ +2h 再触发（同条目 ≤3 次） ├─ 「忽略」→ 本周期静默；下周期重算 └─ 「去完成」→ 档案详情 → 补录 / 标记完成 → 写回计划日

设计对齐：参考 iOS HIG 通知授权与本地通知限制；复杂操作承载对齐 Material Design · Bottom Sheet，减少跨端手势冲突。

💡 本文档用于评审对齐实体、状态字段与边界；首轮评审通过率约 80%，后续迭代补全：稍后次数上限、删宠归档、多宠推送队列长度等。

🔍 竞品分析摘要 · 宠伴

围绕 JD「竞品分析 / 交互模式提炼」整理简要对比，突出差异化设计决策来源。

横向对比矩阵：宠伴 vs 多数云端宠物 App（桌面调研归纳）
对比维度 宠伴 多数云端宠物 App
提醒灵活性 按宠物独立提醒策略；稍后 / 忽略粒度到单次事项 常见为账号级或设备推送，模板相对固定，较少支持「单宠单策略」组合
隐私 / 数据路径 默认本地存储 + 可选自建备份；OCR 可走离线 多为云账号体系，数据路径依赖服务端，隐私路径相对单一
录入方式 表单 + 本地 OCR 降低疫苗本录入成本 以手动 / 扫码为主，OCR 往往不是主线能力

关键发现 → 设计决策

  • 发现：多数竞品未很好支持「多宠独立提醒 + 单次粒度安抚（稍后/忽略）」。
  • 决策：优先交付按宠独立提醒与本地 OCR，把差异化写进 PRD 状态表与边界条件，避免做成又一个通用日历。
