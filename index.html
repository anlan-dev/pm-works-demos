PM核心能力 · 需求发现

用户访谈 + 行为数据 + 差评挖掘三管齐下，从"用户说的"和"用户做的"两个维度交叉验证需求真伪

🔍
需求发现方法论
从用户行为数据和真实反馈中定义需求，而非从技术能力出发找场景
1
方法论框架
三管齐下：①用户深度访谈（挖掘隐性需求）②SQL行为数据（验证用户真实行为）③应用商店差评分析（发现竞品未满足的需求）。交叉验证"用户说的"和"用户做的"，过滤伪需求。
2
案例：AI健身产品线 · 行为数据驱动
通过SQL拆解AI纠错功能全链路（上传视频→触发分析→查看报告→分享）各环节转化数据，发现「查看报告→分享」环节跳出率60%——用户完成训练后才看到纠错报告，错误动作模式已经形成。结合用户反馈验证：用户不是不想纠错，而是反馈来得太晚。由此定义"将反馈时机从训练后前置到训练中"的核心需求。
3
案例：宠伴 · 访谈+差评挖掘
20位宠物主深度访谈+应用商店差评分析，量化定义核心问题——70%用户每月遗忘疫苗接种，83%将隐私顾虑置于成本之上。主动砍掉社区/商城模块，聚焦轻量健康记录，确定"隐私优先+零成本"的产品定位。
4
案例：会议·反刍舱 · 场景观察
访谈PM/项目协作人群，发现会后整理纪要平均耗时15-20分钟，漏项率约30%。核心矛盾不是"没有工具"，而是"工具太重"——用户需要的是"这一场会议的产出可被追踪"。明确不做大而全的协作套件，聚焦最小闭环。

PM核心能力 · 竞品分析

多维度结构化对比 + LLM辅助批量分析竞品更新日志，提炼可落地的产品决策

⚔️
竞品分析方法论
按迭代趋势/共性方向/差异化方向/用户痛点四维度输出结构化报告
1
方法论框架
四维度结构化对比：①迭代趋势（近12个月更新方向）②共性功能（行业标配）③差异化（竞品独有）④用户痛点（差评高频词）。用LLM辅助批量分析竞品更新日志和用户评论，输出可落地的产品决策。
2
案例：AI高尔夫产品线 · 4款竞品深度分析
用DeepSeek/ChatGPT批量分析Golfcam、18Birdies、The Grint、Golfshot近12个月App Store更新日志和用户评论。关键发现：行业趋势从"记录工具"转向"社交分享"，用户核心需求从"看数据"升级为"晒成绩"。直接推动产品决策：海报模板首批设计6套（后优化为3套精选）、历史记录提升为首页一级入口、海报尺寸适配微信朋友圈。
3
案例：宠伴 · 竞品功能对比
对比4款宠物管理类App的疫苗提醒功能、隐私策略、付费模式，发现现有工具重社交商城、轻健康记录，隐私保护均弱于用户预期。由此确定"本地存储+零服务器"的技术差异化路线，把差异化写进PRD。
4
案例：跨境电商 · 市场格局分析
调研跨境电商全链路痛点（需求模糊、市场信息碎片化、选品靠经验、文案多语言难、合规风险高、运营策略缺失），对比传统Chatbot与Multi-Agent协作的效率差异。发现用户"不知道该问什么"比"找不到答案"更致命，由此设计"6-Agent全链路协作"范式——从需求解析到运营策略，6个Agent各司其职，Orchestrator路由调度，系统主动推送结构化信息卡片。

PM核心能力 · 数据分析

SQL埋点分析 + 漏斗转化 + 数据验证闭环，用数据驱动需求优先级决策和效果验证

📊
数据分析方法论
定义指标→埋点设计→数据采集→漏斗分析→验证闭环
1
方法论框架
数据分析闭环：①定义核心指标（转化率/NPS/留存等）②埋点事件设计（事件名/触发时机/关键参数）③SQL数据采集与漏斗分析④A/B测试验证⑤数据反向驱动需求优先级。
2
案例：高球记 · 全模块埋点体系
基于原型Demo定义全模块埋点事件体系（共26个事件覆盖11个页面），其中单杆轨迹模块8个事件（首页点击→导入视频→相册/相机选择→轨迹颜色→重播/导出）为核心工作。定义事件名、触发时机、关键参数（trace_color颜色选择等），输出埋点验收标准。用SQL验证单杆轨迹核心漏斗（导入→导出）的事件触发率与参数完整性。
3
案例：宠伴 · A/B验证体系
建立"假设→实验→指标"验证体系：提醒配置意愿指数（任务完成率0.35+配置意愿量表0.35+主观耗时0.30）+13%，记录耗时缩短约60%，种子用户NPS +12→+38，转化率42%，7日留存44%。[内部测试·种子用户轮次验证]
4
案例：PRD Quality Gate · 效果测评
~5000字PRD实测850ms响应，检出6/19项合规项定位13处遗漏，验证System Prompt直写规则优于RAG检索的适用边界。用数据证明"确定性审核场景用Prompt直写、开放性场景用RAG"的技术选型结论。[内部实测·单次PRD样本]

交付成果：
  • 并行管理3条产品线（健身/高尔夫/视频分析），P0交付率100%
  • 首页功能合并方案（体态评估+松解工具→"训练"）内部走查一轮过关，沟通成本大幅降低
  • P0/P1/P2优先级体系让资源分配从"事后解释"变为"事前共识"，团队对齐效率显著提升
  • 在资源受限的创业环境下，通过优先级体系确保核心交付线不失守，突发需求不再打乱已有节奏

背景

2
问题定义 · 核心判断
基于个人积累的19项PRD质量检查清单，识别到这部分工作可通过AI自动化——规则确定性强，适合System Prompt直写，无需RAG知识库。核心判断：PRD检查是确定性规则检查→System Prompt比RAG更稳定可控。
为什么选System Prompt直写而非RAG：RAG适合"答案在知识库里、需要检索"的场景（如客服问答）。但PRD检查的规则是固定的（"用户故事是否包含验收标准""异常流程是否覆盖"等），不需要检索外部知识，只需要把规则"教"给LLM。用RAG反而会引入检索噪声——检索到不相关的规则片段会干扰判断。实测验证：System Prompt方案的遗漏检出密度（13处/5000字）优于RAG方案（8处/5000字）。
为什么Token消耗更低：选择System Prompt直写的另一个被低估的优势是Token成本效率。RAG方案每次审核都要经过"知识库检索→上下文拼接→LLM推理"三步，其中检索和拼接会额外消耗Token。而System Prompt方案将19项规则直接内嵌在Prompt中，单次审核Token消耗比RAG方案低约60%。对于高频使用场景（如每日多份PRD审核），Token累积节省量级可观。这体现了PM的成本意识——不仅考虑"能不能做"，更考虑"规模化后的单位成本"。
为什么Temperature设0.2而非0：Temp=0会导致输出完全确定，对于"判断PRD是否有问题"这种需要一定推理的任务，完全确定性会降低判断灵活性。Temp=0.2在"可控"和"灵活"之间取平衡——相同PRD多次审核结果一致性>95%，同时保留对模糊表述的推理能力。
为什么用四维拆解（输入/逻辑/异常/评估）：PRD遗漏通常不是随机分布的，而是集中在特定维度。四维拆解基于MECE原则：输入侧检查"有没有说清楚要什么"，逻辑侧检查"逻辑是否自洽"，异常侧检查"边界情况有没有覆盖"，评估侧检查"怎么验证做对了"。实测13处遗漏中，异常侧占7处（54%），验证了"PRD最常遗漏异常流程"的行业共识。

双模式设计：规则引擎保底（离线可用）+AI创意模式（联网增强）。AI失败自动降级，离线兜底触发率<5%。多平台LLM支持。
为什么双模式设计：用户拖延时往往是情绪低落或焦虑状态，此时如果AI服务不可用（网络差/服务器限流），会加重挫败感导致直接放弃。规则引擎保底确保"任何时候打开都能用"——即使AI挂了，也能按预设规则拆解最小行动。AI增强在联网时提供更好的个性化诊断，但产品可用性不依赖它。
为什么用倒计时作为执行模式：拖延的本质是"启动成本高"，不是"不想做"。倒计时创造"有限承诺"的心理框架——"只需要坚持25分钟"，降低启动门槛。这是番茄钟的行为心理学原理：把模糊的"去做吧"转化为具体的"现在开始倒计时"，用外部时间约束替代内部意志力。
为什么AI诊断拖延原因而非直接给方案：如果直接给方案（"去做吧"），和用户的自我对话没有区别。诊断拖延原因能帮用户看到自己看不到的心理障碍——完美主义倾向、任务粒度太大、害怕失败、缺乏意义感。这些是AI比人类朋友更擅长的，因为没有社交压力，用户愿意对AI坦诚。

(b) 技术方案权衡
为什么用 Canvas 本地计算而非服务端 API：
维度Canvas 本地计算（当前方案）服务端 API（如 Google Vision API）

离线可用是否
决策逻辑：Demo 阶段优先验证产品逻辑而非模型精度。Canvas 方案可以快速验证"评分透明化"的产品假设——如果连基础像素统计得出的评分用户都愿意信任并采纳，那产品方向就对了，后续升级模型是工程问题。如果用户根本不信任 AI 评分，再好的模型也没用。这个决策逻辑和我在宠伴选择 Tesseract.js 本地 OCR 的逻辑一致——PM 不能等到"完美方案"才做用户验证。
(c) 四维评分权重设计的依据
核心原则：权重反映产品假设，Demo 阶段未做用户校验。
• 清晰度 35%：最基础的筛选条件——模糊照片在任何场景下都不可用。权重最高，但评分函数设计为"够用即可"（30-95 分区间，普通清晰度即可得 60+），而非追求极致锐度
• 构图 35%：画面结构的核心判断——三等分法则、中心对称、引导线等。与清晰度并列最高权重，因为构图是"拍得好不好"的核心维度，也是手机摄影用户最常提及的评判标准
• 色彩 20%：色彩丰富度和和谐度——高饱和≠好照片，但极度单调的灰暗画面通常不受欢迎。中等权重，因为色彩是"锦上添花"而非"雪中送炭"
• 亮度 10%：曝光是否合理——过曝和欠曝都扣分，但曝光问题可以通过增强功能弥补（而模糊或构图差无法弥补），因此权重最低
PM 视角：权重设计本身就是产品决策——它代表了"你认为什么样的照片是好的"。如果权重设置和用户审美不一致，用户永远不会信任 AI 的推荐。Demo 阶段用固定权重快速验证方向，生产环境中应支持用户自定义权重（"我更在意构图"→ 构图权重上调）。
(d) 增强效果用 CSS Filter 而非真正 AI 增强的设计意图
核心问题：为什么不用 Remini 式的真实 AI 增强？
• 这是 PM Demo，不是产品交付：Demo 的目的是验证"让用户对比原图 vs 增强"这个交互概念是否有价值，以及"增强开关独立可控"是否被用户需要——这两点用 CSS Filter 模拟完全足够
• 真实的 AI 增强（超分辨率/去噪/色彩重建）需要 GPU 推理：Web 端可以用 TensorFlow.js 或 ONNX Runtime Web 跑轻量模型，但会增加 5-15MB 的模型加载时间和 2-5 秒的推理延迟——Demo 阶段的用户体验损害远大于收益
• CSS Filter 本身就是产品洞察：降噪=对比度微调、锐化=亮度微调、色彩增强=饱和度提升——这些滤镜模拟的不是"AI 做了什么"，而是"用户期望 AI 做到的效果"。Demo 验证的是需求层面而非技术层面
PM 视角：一个好的 Demo 不需要完美的技术实现——它需要让用户"看到"产品概念。如果用户不理解"AI 增强"意味着什么，给他看再精细的超分辨率结果也没用。滑块对比 + 独立开关的设计本身比 AI 模型的精度更重要——它解决了"用户不信任 AI 修图"这个比"AI 修得不够好"更根本的问题。

没有使用LangChain。本项目采用原生搭建方案，技术栈如下：
• LLM层：OpenAI GPT-4o（主模型）+ GPT-4o-mini（轻量子任务），通过OpenAI API直接调用
• 编排层：自研Orchestrator路由器，基于Node.js实现MCP服务总线，负责请求分发、结果聚合、错误重试
• Agent层：每个Sub Agent是独立的System Prompt + 工具定义，通过Function Calling与MCP Server交互
• 协议层：MCP（Model Context Protocol）作为Agent与外部工具/数据源的标准接口
• 前端：单页HTML应用，结构化卡片渲染
为什么不用LangChain：LangChain是通用框架，抽象层多、调试链路长。本项目Agent数量固定（6个）、协作模式明确（流水线+并行），用原生代码直接编排更轻量、可控、易调试。LangChain更适合需要动态Agent创建或复杂RAG检索的场景。
(c) 架构选择：原生Agent vs 嵌入OpenClaw
两种方式的对比：
维度原生搭建Agent嵌入OpenClaw+MCP
开发成本高（需自建编排、错误处理、状态管理）低（平台提供基础设施）
控制精度高（每个环节可精确控制Prompt、参数、输出格式）中（受平台抽象层约束）
调试效率高（链路透明，可逐环节Debug）中（平台黑盒，需通过日志排查）
扩展性中（需手动扩展编排逻辑）高（平台支持动态Agent注册）
适合场景Agent数量固定、协作模式明确、需要精确控制输出Agent数量动态、需要快速原型、团队协作开发
本项目选择原生搭建的原因：①6个Agent的协作模式是固定的流水线+并行，不需要动态创建Agent；②跨境电商场景对输出格式要求严格（结构化卡片），需要精确控制每个Agent的输出Schema；③作为PM个人项目，需要深入理解Agent编排的每个细节，而不是把问题丢给平台黑盒。如果未来Agent数量扩展到10+或需要动态编排，会考虑迁移到OpenClaw/AutoGen等平台。
(d) 为什么Sub Agent模式适合跨境电商场景
核心原因：任务可拆解、领域可隔离、失败可定位。
• 任务可拆解：跨境电商的业务链路天然分为6个独立阶段（需求→调研→选品→文案→合规→运营），每个阶段有明确的输入/输出，适合拆分为独立Agent
• 领域可隔离：每个Agent的System Prompt和工具集完全不同（调研Agent需要搜索工具，合规Agent需要法规库，文案Agent需要翻译工具）。如果用单个大Agent，Prompt会变得极其臃肿，上下文窗口会被无关信息占用
• 失败可定位：Sub Agent模式下，如果最终结果有问题，可以通过检查每个Sub Agent的输出精确定位是哪个环节出了问题。单Agent模式下所有逻辑混在一起，难以定位
对比单Agent方案：如果用单个GPT-4o完成全部6个环节，需要一次性输入所有工具定义和业务规则，Prompt长度会超过4000 tokens，导致注意力分散、输出质量下降。拆分为6个Sub Agent后，每个Agent的Prompt控制在500-800 tokens，专注度更高。
(e) Main Agent与Sub Agent的通讯方式
通讯架构：Main Agent（Orchestrator）通过MCP服务总线向Sub Agent发送结构化请求，Sub Agent处理后返回结构化JSON响应。
Sub Agent对Main Agent的输出结构：针对通过筛选和未通过筛选两种情况，定义了不同的输出Schema：
// 通过筛选的输出结构 { "status": "pass", "agent": "需求解析Agent", "intent": { "category": "选品", "keywords": ["蓝牙耳机","TWS"], "confidence": 0.95 }, "structured_data": { "budget": "200-500元", "target_market": "东南亚", "quantity": 100 }, "next_agent": "市场调研Agent", "latency_ms": 180 } // 未通过筛选的输出结构 { "status": "fail", "agent": "合规审核Agent", "reason": "商品涉及CE认证缺失", "severity": "high", "suggestion": "建议补充CE认证文件或更换供应商", "blocked": true, "fallback": "运营策略Agent（生成合规整改建议）" }
设计决策：通过/失败两种输出结构的核心差异在于：pass状态携带next_agent字段引导流转，fail状态携带suggestion和fallback字段提供补救路径。这样Main Agent不需要理解业务逻辑，只需根据status字段决定路由——pass则转发给next_agent，fail则执行fallback或终止。

底层逻辑：Harness 五步法
所有 AI 项目遵循同一思维链路——从用户任务出发，经过评测验证，用数据驱动迭代
功能点 → AI技术方案 → 评测集+MECE标准 → 跑评测找链路问题 → 改链路重测 → 展示分数改善

2选择AI技术方案（Prompt Chain / RAG / Workflow / Agent Team），解释为什么不用更复杂/更简单的
3设计评测集 + MECE评分标准，目的是发现链路问题而非证明"AI很厉害"
4跑评测集，定位链路节点具体问题，归因到 prompt / query rewrite / context / routing / fallback
5针对低分维度迭代对应节点，用同一评测集复测，观察指标改善

Token 经济学：省钱比换模型更重要
LLM 调用成本是 AI 产品的隐形定价锚——本月多个压缩项目爆火
趋势：chopratejas/headroom 实现 60-95% token 节省，DeusData/codebase-memory-mcp（Rust 引擎）减少 99.2% token 消耗，LMCache 加速 LLM KV 缓存——GitHub 上"省钱"类项目正在批量涌现。这不是简单的"用更便宜的模型"，而是在不牺牲质量的前提下，通过智能压缩和缓存降低调用成本。行业共识：Token 成本控制是 AI 产品商业化的前提。
我的实践：跨境电商 Agent 通过**意图路由压缩**减少无效调用——先由 Intent Parser 判断请求类型，再路由到对应 Sub Agent，而非每次全链路跑 6 个 Agent（降低约 40% token 消耗）。PRD Quality Gate 选 System Prompt 直写而非 RAG——不仅因为规则确定性强，更因为每次 RAG 检索会引入额外的 token 成本（知识库检索 + 上下文拼接），且检索噪声可能反向损害判定准确率。实测：System Prompt 方案 Token 消耗比 RAG 方案低约 60%。
PM启示：PM 在 AI 产品设计中必须有"Token 成本意识"——不是工程师独自操心的事。在产品设计阶段就应该问：这里能否用意图路由减少无效调用？能否用缓存避免重复推理？Chunk Strategy 和 Prompt 长度是否经过成本核算？用户不会为"能用"买单，但会为"既好用又省钱"买单。

Agent 安全：26.1% 技能含漏洞，安全审计从可选项变必选项

我的实践：跨境电商 Agent 系统内置 Compliance Agent（合规审核模块）——设计思路上与 SkillSpector 的安全审计理念一致：每个 Sub Agent 的输出不是直接交付用户，而是经过 Compliance Agent 的规则检查。PRD Quality Gate 设计了"三重护栏"机制：操作确认（护栏①）→ 过程可解释（护栏②）→ 结果可修正（护栏③），确保 AI 输出始终在人的最终控制下。
