为什么是AI产品经理？因为我相信，AI时代的产品经理需要同时理解"人的需求"和"AI的能力边界"。语言学训练让我擅长从用户访谈中提取真实需求，而8个AI产品Demo的实践让我建立了对Prompt Engineering、RAG、Agent架构的直觉判断。我不只是"会用AI工具"，而是能定义"什么场景用什么AI能力"，以及"AI失败时如何降级"。

作品集的核心逻辑：从宠伴（本地优先的宠物健康台账）到GlobalFUN（6-Agent跨境电商协作系统），我展示了AI产品设计的三个层次——需求层（用户调研→需求分层→竞品分析）、能力层（AI能力边界判断→Prompt设计→Agent架构）、体验层（交互设计→信息架构→响应式适配）。每个项目都包含完整的用户调研、竞品分析和数据埋点设计，而不是"为了用AI而用AI"。

技术实践：我用Cursor（Vibe Coding）从零搭建了所有Demo，验证了"文科背景PM也能交付可交互产品"。在跨境电商项目中，我设计了MCP服务总线+6-Agent协作架构，覆盖需求解析→市场调研→选品分析→文案生成→合规审核→运营策略的完整闭环。在PRD Quality Gate中，我建立了19项检查规则的自动化审查系统，用Temperature 0.2控制幻觉率。

前沿观察：我持续关注AI Agent架构、MCP/A2A协议生态、Vibe Coding趋势、WWDC 2026苹果AI战略等前沿动态，并在作品集的"前沿观察"部分记录了我的思考。我相信，AI产品经理的核心竞争力不是"会写Prompt"，而是"能定义AI产品的边界和降级策略"。

PM核心能力 · 需求发现

用户访谈 + 行为数据 + 差评挖掘三管齐下，从"用户说的"和"用户做的"两个维度交叉验证需求真伪

🔍
需求发现方法论
从用户行为数据和真实反馈中定义需求，而非从技术能力出发找场景
1
方法论框架
三管齐下：①用户深度访谈（挖掘隐性需求）②SQL行为数据（验证用户真实行为）③应用商店差评分析（发现竞品未满足的需求）。交叉验证"用户说的"和"用户做的"，过滤伪需求。
2
案例：EasyRight AI · 行为数据驱动
通过SQL拆解AI纠错功能全链路（上传视频→触发分析→查看报告→分享）各环节转化数据，发现「查看报告→分享」环节跳出率60%——用户完成训练后才看到纠错报告，错误动作模式已经形成。结合用户反馈验证：用户不是不想纠错，而是反馈来得太晚。由此定义"将反馈时机从训练后前置到训练中"的核心需求。
3
案例：宠伴 · 访谈+差评挖掘
20位宠物主深度访谈+应用商店差评分析，量化定义核心问题——70%用户每月遗忘疫苗接种，83%将隐私顾虑置于成本之上。主动砍掉社区/商城模块，聚焦轻量健康记录，确定"隐私优先+零成本"的产品定位。
4
案例：会议·反刍舱 · 场景观察
访谈PM/项目协作人群，发现会后整理纪要平均耗时15-20分钟，漏项率约30%。核心矛盾不是"没有工具"，而是"工具太重"——用户需要的是"这一场会议的产出可被追踪"。明确不做大而全的协作套件，聚焦最小闭环。

PM核心能力 · 竞品分析

多维度结构化对比 + LLM辅助批量分析竞品更新日志，提炼可落地的产品决策

⚔️
竞品分析方法论
按迭代趋势/共性方向/差异化方向/用户痛点四维度输出结构化报告
1
方法论框架
四维度结构化对比：①迭代趋势（近12个月更新方向）②共性功能（行业标配）③差异化（竞品独有）④用户痛点（差评高频词）。用LLM辅助批量分析竞品更新日志和用户评论，输出可落地的产品决策。
2
案例：高球记 · 4款竞品深度分析
用DeepSeek/ChatGPT批量分析Golfcam、18Birdies、The Grint、Golfshot近12个月App Store更新日志和用户评论。关键发现：行业趋势从"记录工具"转向"社交分享"，用户核心需求从"看数据"升级为"晒成绩"。直接推动产品决策：海报模板首批设计6套（后优化为3套精选）、历史记录提升为首页一级入口、海报尺寸适配微信朋友圈。
3
案例：宠伴 · 竞品功能对比
对比4款宠物管理类App的疫苗提醒功能、隐私策略、付费模式，发现现有工具重社交商城、轻健康记录，隐私保护均弱于用户预期。由此确定"本地存储+零服务器"的技术差异化路线，把差异化写进PRD。
4
案例：跨境电商 · 市场格局分析
调研跨境电商全链路痛点（需求模糊、市场信息碎片化、选品靠经验、文案多语言难、合规风险高、运营策略缺失），对比传统Chatbot与Multi-Agent协作的效率差异。发现用户"不知道该问什么"比"找不到答案"更致命，由此设计"6-Agent全链路协作"范式——从需求解析到运营策略，6个Agent各司其职，Orchestrator路由调度，系统主动推送结构化信息卡片。

PM核心能力 · 数据分析

SQL埋点分析 + 漏斗转化 + 数据验证闭环，用数据驱动需求优先级决策和效果验证

📊
数据分析方法论
定义指标→埋点设计→数据采集→漏斗分析→验证闭环
1
方法论框架
数据分析闭环：①定义核心指标（转化率/NPS/留存等）②埋点事件设计（事件名/触发时机/关键参数）③SQL数据采集与漏斗分析④A/B测试验证⑤数据反向驱动需求优先级。
2
案例：高球记 · 全模块埋点体系
基于原型Demo定义全模块埋点事件体系（共26个事件覆盖11个页面），其中单杆轨迹模块8个事件（首页点击→导入视频→相册/相机选择→轨迹颜色→重播/导出）为核心工作。定义事件名、触发时机、关键参数（trace_color颜色选择等），输出埋点验收标准。用SQL验证单杆轨迹核心漏斗（导入→导出）的事件触发率与参数完整性。
3
案例：宠伴 · A/B验证体系
建立"假设→实验→指标"验证体系：提醒配置意愿指数（任务完成率0.35+配置意愿量表0.35+主观耗时0.30）+13%，记录耗时缩短约60%，种子用户NPS +12→+38，转化率42%，7日留存44%。
4
案例：PRD Quality Gate · 效果测评
~5000字PRD实测850ms响应，检出6/19项合规项定位13处遗漏，验证System Prompt直写规则优于RAG检索的适用边界。用数据证明"确定性审核场景用Prompt直写、开放性场景用RAG"的技术选型结论。

产品方法论

🔄
底层逻辑：Harness 五步法
所有 AI 项目遵循同一思维链路——从用户任务出发，经过评测验证，用数据驱动迭代
功能点 → AI技术方案 → 评测集+MECE标准 → 跑评测找链路问题 → 改链路重测 → 展示分数改善
1从具体用户任务出发，不是从模型能力出发
2选择AI技术方案（Prompt Chain / RAG / Workflow / Agent Team），解释为什么不用更复杂/更简单的
3设计评测集 + MECE评分标准，目的是发现链路问题而非证明"AI很厉害"
4跑评测集，定位链路节点具体问题，归因到 prompt / query rewrite / context / routing / fallback
5针对低分维度迭代对应节点，用同一评测集复测，观察指标改善
🚦
六道关卡（Stage Gate 1-6）
设计任何 AI 功能前，逐一验证这六关——任一关不通过就暂停
G关卡检查什么我的实际应用
1场景与需求用户是谁？失败成本？"练错无人纠正"→训练效果差+受伤风险
2AI必要性为什么不用规则/人工？PRD检查=确定规则→System Prompt比RAG更稳定
3链路可落地输入→AI→验证→输出→人确认→反馈三重护栏：操作确认→过程可解释→结果可修正
4评测可设计怎么判断AI好坏？四维拆解+人工对照验证
5Badcase迭代出错怎么归因改？13处遗漏→按维度归类→优化Prompt
6证据边界能写/不能写什么？实测数据可讲，不编造未上线效果

AI Agent 架构与落地模式
2026年Agent从"单工具调用"走向"多Agent协作工程化"
趋势：2026年AI Agent架构发生质变。Anthropic于2月推出Claude Code Agent Teams——team lead与teammates独立上下文并行协作，16个并行Claude约2000 sessions构建10万行C编译器。Microsoft于6月发布Copilot Studio Agent Orchestrator，支持多Agent智能调度与递归任务执行。Agent不再是"调用工具"，而是"Agent间协作"的工程问题。
我的实践：跨境电商AI Agent中设计了MCP服务总线+6-Agent（需求解析/市场调研/选品分析/文案生成/合规审核/运营策略）全链路协作架构，实现结构化卡片输出与多Agent并行协作。冷启动助手的Prompt模板本质上是轻量级Agent定义。
PM 启示：PM需要定义的不只是"用什么工具"，还有"Agent间如何委托、失败时谁负责、信任如何传递"。多Agent系统的PM需要设计：任务分解策略、Agent能力边界、失败升级机制、结果聚合逻辑。

我的实践：建了12个标准化Prompt文件，覆盖6步PM工作流。PRD Quality Gate本质上是一个Prompt工程项目——把19项检查规则写进System Prompt，Temperature 0.2控制幻觉。跨境电商的6-Agent协作架构中，每个Agent都有独立的系统指令和工具调用策略。
PM 启示：未来PM的"原型能力"包括：用Prompt快速验证产品假设、用AI生成PRD初稿、用AI辅助评审自查。更重要的是，PM需要设计Agent的"人格"——它的能力边界、失败行为、与用户交互的方式。

AI Native 产品设计思维
WWDC 2026 Siri AI：AI Native设计的典型案例
趋势：2026年AI Native产品形态加速涌现。Perplexity不是"搜索引擎+AI"，而是"AI回答引擎"；v0不是"设计工具+AI"，而是"自然语言→界面"。WWDC 2026的Siri AI更是典型案例——不是"语音助手+AI"，而是"操作系统级Agent"，能跨应用访问照片、邮件、短信、文件，结合网络信息综合分析，完成复杂任务。产品形态本身由AI能力定义，而非在现有功能上叠加AI。
我的实践：PRD Quality Gate是典型的AI Native设计——它不是"PRD模板+AI检查"，而是"AI自动审查PRD质量"这个全新产品形态。宠伴的"先数据后功能"策略也是AI Native思路：先用本地OCR建立数据壁垒，再基于数据提供AI健康报告。
PM 启示：AI Native产品设计的核心不是"AI能做什么"，而是"用户的工作流中，哪些环节可以被AI重新定义"。PM应该从用户任务链出发，找到AI介入的最佳节点，而不是反过来从技术能力出发找场景。

端侧AI与隐私计算的产品化机遇
WWDC 2026：Apple Foundation Models让端侧AI真正可用
趋势：WWDC 2026（2026年6月8日）Apple Intelligence重大升级。Apple Foundation Models（AFM）包括本地模型AFM Core/Core Advanced和云端模型AFM Cloud，采用"稀疏方法"让更大规模模型在设备端高效运行。Siri AI能访问用户设备中的照片、邮件、短信、文件，结合网络信息综合分析，具备"屏幕理解"能力——识别当前显示的网页、文档、图片并执行操作。所有任务执行均采取高强度隐私保护，只在设备端或专用云计算服务器端进行。苹果与Google Gemini合作但运行在自有芯片和基础设施上。
我的实践：宠伴选择Tesseract.js本地OCR而非云端API，正是因为访谈中83%用户明确拒绝上传疫苗凭证。会议舱转写也优先端侧处理。这些不是"技术选型"，而是"产品策略"——隐私优先本身就是卖点。
PM 启示：端侧AI让"隐私优先"不再是体验妥协，而是可以做到"既安全又好用"。PM在设计AI功能时，应该先评估：这个功能能否端侧完成？如果能，就不应该要求用户上传数据。这不仅是合规要求，更是用户信任的基石。

多Agent协作协议生态
MCP + A2A + ANP 三大协议构建Agent通信基础设施
趋势：2026年Agent通信协议生态快速成熟。三大协议定位清晰：MCP管纵向（Agent↔工具访问）、A2A管横向（Agent↔Agent协作）、ANP管网状（大规模调度）。MCP由Anthropic开源，SDK月下载量9700万次，公开服务器21,962个。A2A由Google捐赠给Linux基金会，清华大学2026年6月发布A2A研究报告，将其定义为"AI时代数字委托协议与组织基础设施"。微信与华为、小米、OPPO、vivo等手机厂商合作推出A2A助手能力，标志着Agent互联协议正式进入商用阶段。
PM 启示：PM需要理解三大协议的产品含义：MCP决定"Agent能调用什么工具"，A2A决定"Agent间如何委托和协作"，ANP决定"大规模Agent网络如何组织和发现"。在设计多Agent系统时，PM需要明确：哪些任务用MCP调用工具，哪些任务用A2A委托给其他Agent，以及如何设计信任传递和失败升级机制。

WWDC 2026 与苹果AI战略
Siri AI从语音助手进化为操作系统级Agent
趋势：WWDC 2026（2026年6月8日）苹果发布全新Apple Intelligence架构。Siri AI不再是传统"语音指令工具"，而是进化为操作系统级Agent——具备上下文理解能力、屏幕感知能力、跨应用任务执行能力。用户可以让Siri AI访问照片、邮件、短信、文件，结合网络信息综合分析，完成"导航去Jeff家"这样的复杂任务。Apple Foundation Models（AFM）包括本地模型AFM Core/Core Advanced和云端模型AFM Cloud，采用"稀疏方法"让更大规模模型在设备端高效运行。苹果与Google Gemini合作但运行在自有芯片和基础设施上，强调"隐私与信任"作为AI竞争的核心武器。
PM 启示：WWDC 2026展示了AI Native产品设计的最高水平——不是"语音助手+AI"，而是"操作系统级Agent"。PM应该学习苹果的产品思路：从用户任务链出发，找到AI介入的最佳节点；隐私优先不是妥协，而是竞争优势；端侧AI让"既安全又好用"成为可能。
