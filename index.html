独立项目

🔍
PRD Quality Gate · AI自动化审核 已验证
独立产品 · 2025.12-至今
1
背景
团队PRD评审依赖人工逐条检查，经常遗漏模糊点和歧义描述，导致开发阶段反复返工。
2
问题定义 · 核心判断
基于个人积累的19项PRD质量检查清单，识别到这部分工作可通过AI自动化——规则确定性强，适合System Prompt直写，无需RAG知识库。核心判断：PRD检查是确定性规则检查→System Prompt比RAG更稳定可控。
3
产品架构
平台：Dify Chatflow · 模型：mimo-v2-flash · Temperature 0.2（控制幻觉）· 审核四维：输入侧/逻辑侧/异常侧/评估侧 · 三重护栏：操作确认→过程可解释→结果可修正
4
功能 + Harness底层逻辑
  • 用户提交PRD→AI逐项检查19项规则→生成结构化审核报告
  • 每项"未通过"附带具体引用和判断依据（过程可解释）
  • 用户可手动修改AI判断（结果可修正）
  • 方案权衡采用"效果/成本/周期矩阵"进行选型决策
Harness 五步链路（本项目）
1功能点：PRD评审中人工逐条检查遗漏模糊点和歧义描述
2AI方案：System Prompt直写19项规则 + Dify Fixed Workflow + Temp 0.2
3评测+MECE：高球记V5.0 PRD(~5000字)→四维拆解（输入/逻辑/异常/评估）→每项0/1判定
4跑评测：检出6/19合规+13处遗漏→人工对照验证AI判断准确性
5改链路：针对漏判维度优化System Prompt描述→补第20项规则→同PRD复测验证改善
5
用户流程
提交PRD→确认输入(护栏①)→AI逐项检查19条→输出报告+依据(护栏②)→确认/修改(护栏③)
6
实测结果
~5000字生产级PRD ~850ms响应时间 6/19合规项 13处定位遗漏
核心结论
验证了System Prompt直写规则比RAG更适用于确定性审核场景。项目已上线交付团队使用。

产品方法论

🔄
底层逻辑：Harness 五步法
所有 AI 项目遵循同一思维链路——从用户任务出发，经过评测验证，用数据驱动迭代
功能点 → AI技术方案 → 评测集+MECE标准 → 跑评测找链路问题 → 改链路重测 → 展示分数改善
1从具体用户任务出发，不是从模型能力出发
2选择AI技术方案（Prompt Chain / RAG / Workflow / Agent Team），解释为什么不用更复杂/更简单的
3设计评测集 + MECE评分标准，目的是发现链路问题而非证明"AI很厉害"
4跑评测集，定位链路节点具体问题，归因到 prompt / query rewrite / context / routing / fallback
5针对低分维度迭代对应节点，用同一评测集复测，观察指标改善
🚦
六道关卡（Stage Gate 1-6）
设计任何 AI 功能前，逐一验证这六关——任一关不通过就暂停
G关卡检查什么我的实际应用
1场景与需求用户是谁？失败成本？"练错无人纠正"→训练效果差+受伤风险
2AI必要性为什么不用规则/人工？PRD检查=确定规则→System Prompt比RAG更稳定
3链路可落地输入→AI→验证→输出→人确认→反馈三重护栏：操作确认→过程可解释→结果可修正
4评测可设计怎么判断AI好坏？四维拆解+人工对照验证
5Badcase迭代出错怎么归因改？13处遗漏→按维度归类→优化Prompt
6证据边界能写/不能写什么？实测数据可讲，不编造未上线效果

前沿观察与行业思考

不止于做——对 AI 产品趋势的跟踪与产品化思考

🤖
AI Agent 架构与落地模式
MCP / Function Calling 正在重新定义 PM 的产品边界
趋势：2025年 AI Agent 从单轮对话走向多工具协作。MCP 标准化了模型调用外部工具的方式，Function Calling 成为 Agent 的"神经系统"。
我的实践：跨境电商 AI Agent 中设计了 MCP 服务总线 + 多 Agent（比价/翻译/物流）协同架构，实现结构化卡片输出。冷启动助手的 Prompt 模板本质上是轻量级 Agent 定义。
PM 启示：Agent 不是万能的——PM 的核心价值是定义"什么场景用什么工具链"，以及失败时的降级策略。
✍️
Prompt Engineering 作为PM核心技能
好的 Prompt = 好的需求文档：结构化、可复现、可测试
我的实践：建了 12 个标准化 Prompt 文件，覆盖 6 步 PM 工作流。PRD Quality Gate 本质上是一个 Prompt 工程项目——把 19 项检查规则写进 System Prompt，Temperature 0.2 控制幻觉。
PM 启示：未来 PM 的"原型能力"包括：用 Prompt 快速验证产品假设、用 AI 生成 PRD 初稿、用 AI 辅助评审自查。
🔒
隐私优先的产品设计
本地优先不是技术选型，是产品策略
我的实践：宠伴用 PWA+本地存储，零服务器依赖。会议舱转写端侧优先，OCR 选本地识别因 83% 用户拒绝云上传。
PM 启示：PM 应在需求阶段就回答：这些数据必须上云吗？不上云用户体验会变差吗？

Vibe Coding 与 PM 原型能力升级
2026年63%的Vibe Coding用户是非工程师——PM的"原型能力"正在被重新定义
趋势：Vibe Coding 从2025年安德烈·卡帕西的概念推文，到2026年成为Collins词典年度词汇，市场规模达47亿美元。Cursor估值293亿美元，92%的美国开发者日常使用AI编程工具。歌手胡彦斌用AI编程做App、非技术创始人两周上线SaaS——"用自然语言描述需求，AI完成从架构到部署的全链路"已成为现实。
我的实践：我用 Cursor 从零搭建了宠伴PWA（含本地OCR、Chart.js看板、双端适配），创作舱6页完整原型也是Vibe Coding产出。作为一个文科背景的PM，我验证了"不懂代码也能交付可交互产品"这个命题。
PM 启示：未来PM的核心竞争力不是"画原型"，而是"描述清楚需求+验证AI产出"。PRD写得越精确，AI生成的结果越可用。PM的新工作流：写PRD → Cursor生成初版 → 人工验收badcase → 迭代，本质上和Harness五步法一致。

MCP：AI Agent 的"USB-C"标准
Model Context Protocol 正在成为AI连接外部系统的统一协议——PM需要理解它的产品含义
趋势：MCP 由 Anthropic 于2024年底开源，2025年底捐赠给 Linux 基金会，2026年已成为行业事实标准——OpenAI、Google、Microsoft、AWS 全部支持。SDK月下载量超9700万次。它解决了N×M集成问题：过去每个AI工具对接每个外部系统都需要定制开发，现在通过一个标准协议即可完成。微软已将MCP集成到Fabric数据平台，Cursor、Claude等工具原生支持MCP Server。
PM 启示：MCP改变了PM思考"AI功能边界"的方式。过去PM问"这个功能AI能不能做"，现在应该问"这个场景需要连接哪些外部系统，MCP Server是否已存在"。在EasyRight实习中，我设计跨境电商AI Agent时用MCP服务总线连接比价/翻译/物流三个Agent——这个架构思路在未来会成为标配。

AI Native 产品设计思维
从"给产品加AI功能"到"用AI重新定义产品形态"——范式正在转换
趋势：2024-2025年大多数产品的AI策略是"在现有功能上叠加AI"——搜索框加个AI摘要、编辑器加个AI续写。2026年的趋势是AI Native：产品形态本身由AI能力定义。Perplexity不是"搜索引擎+AI"，而是"AI回答引擎"；v0不是"设计工具+AI"，而是"自然语言→界面"。Collins词典将Vibe Coding选为年度词汇，标志着"非工程师用自然语言构建软件"进入主流认知。
我的实践：PRD Quality Gate是典型的AI Native设计——它不是"PRD模板+AI检查"，而是"AI自动审查PRD质量"这个全新产品形态。宠伴的"先数据后功能"策略也是AI Native思路：先用本地OCR建立数据壁垒，再基于数据提供AI健康报告。
PM 启示：AI Native产品设计的核心不是"AI能做什么"，而是"用户的工作流中，哪些环节可以被AI重新定义"。PM应该从用户任务链出发，找到AI介入的最佳节点，而不是反过来从技术能力出发找场景。

端侧AI与隐私计算的产品化机遇
当AI推理从云端迁移到设备端，PM需要重新思考数据策略和产品架构
趋势：Apple Intelligence、Google Gemini Nano、高通AI Engine推动端侧AI能力快速增长。2026年主流手机已能本地运行7B参数模型。这意味着OCR、语音转写、图像识别等任务可以在设备端完成，无需上传云端。同时GDPR、中国《个人信息保护法》等法规趋严，用户对数据上传的容忍度持续下降。
我的实践：宠伴选择Tesseract.js本地OCR而非云端API，正是因为访谈中83%用户明确拒绝上传疫苗凭证。会议舱转写也优先端侧处理。这些不是"技术选型"，而是"产品策略"——隐私优先本身就是卖点。
PM 启示：端侧AI让"隐私优先"不再是体验妥协，而是可以做到"既安全又好用"。PM在设计AI功能时，应该先评估：这个功能能否端侧完成？如果能，就不应该要求用户上传数据。这不仅是合规要求，更是用户信任的基石。
